# ADR-0001: Frontend Foundation And FSD Target

- Status: Accepted
- Date: 2026-04-08

## Context

Проект Leader Group реализован как frontend на `Vite + React + TypeScript`.

На текущий момент:

- приложение представляет собой маркетинговый SPA-лендинг;
- роутинг и bootstrap живут в `src/app`;
- страница собирается через `pages` и `widgets`;
- основная живая реализация сосредоточена в `src/widgets`;
- `src/features` и `src/entities` пока почти не используются и выступают как подготовленная архитектурная рамка;
- дизайн уже частично зафиксирован документами в `docs/styles-docs/*`.

Архитектурно в проекте уже декларирован FSD-подход с направлением зависимостей:

`app -> pages -> widgets -> features -> entities -> shared`

Проблема текущего состояния в том, что FSD есть как намерение, но значимая часть контента и предметных моделей всё ещё находится внутри `widgets`.

## Decision

Принимаем следующие решения как базовую линию проекта:

1. Сохраняем текущий стек: `Vite`, `React`, `TypeScript`, `React Router`, `Redux Toolkit`.
2. Сохраняем FSD-подобную структуру как целевую архитектуру проекта.
3. Считаем текущее преобладание `widgets` временным состоянием, а не целевой нормой.
4. Новые предметные модели и связанные с ними данные выносим в `entities` и `features` по мере развития.
5. `widgets` используем прежде всего как слой композиции крупных интерфейсных блоков.
6. `shared` оставляем для переиспользуемой инфраструктуры, базовых UI-примитивов, конфигов и утилит.
7. Redux не расширяем формально "на будущее": используем только при появлении реального общего состояния.

## Consequences

Положительные:

- у проекта сохраняется понятная траектория роста без будущего хаотичного рефакторинга;
- новые агенты получают явное направление развития;
- доменная логика перестаёт накапливаться внутри `widgets`.

Негативные и ограничивающие:

- потребуется постепенный рефакторинг уже существующих `widgets/home/*`;
- часть текущих модулей придётся дробить на более чистые уровни ответственности;
- некоторое время проект будет жить в смешанном состоянии между "текущей реализацией" и "целевой FSD-структурой".

## Implementation Notes

- При новом развитии home-секций сначала проверять, не пора ли выносить данные и модели вниз по слоям.
- Для новых модулей по возможности сразу делать публичный API через `index.ts`.
- Любое значимое уточнение целевой структуры фиксировать новым ADR, а не только правкой в коде.
