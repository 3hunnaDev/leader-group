# ADR-0002: Frontend Testing Tooling

- Status: Accepted
- Date: 2026-04-08

## Context

В проекте уже есть несколько участков с нетривиальным поведением:

- `shared/ui/modal` с focus trap, portal и обработкой клавиатуры;
- `widgets/site-header` с hash-навигацией, адаптивным поведением и реакцией на viewport;
- `widgets/home/projects` с интерактивным переключением;
- части UI, зависящие от browser APIs вроде `matchMedia`, `ResizeObserver`, `scrollTo`, `HashChangeEvent`.

При этом тестовая инфраструктура пока отсутствует. Для текущего размера проекта нужен лёгкий и нативный для Vite стек, без лишней операционной сложности.

## Decision

Базовым стеком для unit/component-тестов принимаем:

- `Vitest`
- `React Testing Library`
- `jsdom`

Также фиксируем правила:

1. Писать тесты по необходимости, а не механически на каждый статический блок.
2. В первую очередь покрывать код с поведением, состоянием и риском регрессий.
3. Тестировать пользовательский результат и observable behavior, а не внутренние детали реализации.
4. Browser APIs мокать локально и прагматично, только когда это нужно конкретному тесту.
5. Не вводить тяжёлый e2e-стек до появления явной продуктовой необходимости.

## Consequences

Положительные:

- стек естественно сочетается с `Vite` и React-компонентами;
- тесты можно писать постепенно, начиная с самых рискованных мест;
- стоимость входа для следующего агента и разработчика остаётся низкой.

Ограничения:

- потребуется настроить базовую test-конфигурацию и scripts;
- для некоторых компонентов понадобятся моки DOM-API;
- анимации и layout-зависимое поведение придётся тестировать выборочно и прагматично.
