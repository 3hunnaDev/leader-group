# AGENTS.md

Этот файл фиксирует рабочий контекст проекта для любого следующего агента. Его цель: быстро восстановить состояние репозитория, не переисследуя проект с нуля.

## Как восстановить контекст

Читать в таком порядке:

1. `AGENTS.md`
2. `README.md`
3. `docs/architecture.md`
4. `docs/styles-docs/ui-template-style-guide.md`
5. `docs/styles-docs/leadergroup-site-copy.md`
6. `docs/styles-docs/specification.md`
7. `docs/adr/README.md`
8. `docs/tasks/backlog.md`
9. `CHANGELOG.md`

## Текущее состояние проекта

- Проект: frontend сайта Leader Group.
- Стек: `Vite`, `React 19`, `TypeScript`, `React Router 7`, `Redux Toolkit`.
- Тип приложения: на текущем этапе это маркетинговый SPA-лендинг.
- Архитектурная цель: FSD-подобная структура `app -> pages -> widgets -> features -> entities -> shared`.
- Фактическое состояние: большая часть живого кода пока находится в `widgets`, а `features` и `entities` подготовлены как точки роста.
- Готовая инфраструктура уже есть в `app`, `shared`, `widgets/site-header`, `shared/ui/modal`.
- Основные content-driven секции страницы собраны в `widgets/home/*`.

## Рабочая стратегия для агента

1. Сначала сохранить текущую визуальную и архитектурную линию, потом расширять проект.
2. Не ломать заявленную FSD-структуру новыми shortcut-решениями.
3. Новую предметную логику, данные и модели по возможности выносить из `widgets` в `entities` и `features`.
4. `widgets` использовать как слой композиции крупных UI-блоков, а не как постоянное место хранения доменных моделей.
5. `shared` оставлять для переиспользуемой инфраструктуры, конфигов, примитивов UI и утилит.
6. Любое заметное архитектурное решение фиксировать через ADR.
7. Любое выполненное изменение добавлять в `CHANGELOG.md`.
8. Любую новую разработческую задачу или крупный следующий шаг добавлять в `docs/tasks/backlog.md`.

## Правила разработки

- Соблюдать направление зависимостей слоёв: `app -> pages -> widgets -> features -> entities -> shared`.
- Для модулей держать явный публичный API через `index.ts`, где это уместно.
- Не добавлять новые глобальные стилистические направления в обход `docs/styles-docs/*`.
- Не менять визуальную ДНК сайта без явного решения: проект уже зафиксирован как clean/minimal/editorial.
- Не размазывать статический контент по случайным компонентам, если он относится к предметной области и может жить в моделях/конфигах.
- Не расширять Redux формально "на будущее": добавлять состояние только при реальной необходимости.

## Правила по тестам

- Базовый стек тестирования для frontend-задач: `Vitest`, `React Testing Library`, `jsdom`.
- Тесты добавлять по необходимости, но приоритетно для кода с поведением:
  - `shared/ui/modal`
  - `widgets/site-header`
  - роутинг, заголовки страницы, hash-навигация
  - интерактивные блоки вроде переключения проектов
- Для статичных секций без логики тесты не обязательны, пока не появляется поведение.
- Предпочтение: тестировать поведение, доступность и публичный результат, а не внутреннюю реализацию.

## Минимальный чеклист перед завершением работы

- `npm run build`
- `npm run lint`
- `npm run lint:styles`
- `npm run format:check`

Если менялась тестовая инфраструктура или добавлялись тесты, запускать также соответствующие test-скрипты.

## Как не потерять контекст

Если агент делает одно из действий ниже, он обязан обновить документацию:

- меняет архитектурный подход: обновить `docs/adr/*`
- добавляет или завершает крупную задачу: обновить `docs/tasks/backlog.md`
- меняет пользовательское или инженерное поведение проекта: обновить `CHANGELOG.md`
- меняет дизайн-правила или тексты-источники: синхронизировать с `docs/styles-docs/*`
